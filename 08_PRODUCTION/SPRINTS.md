# SPRINTS

## Purpose

This document defines how development work is divided into manageable sprints.

A sprint is a short development cycle focused on delivering a complete set of features.

---

# Sprint Philosophy

Every sprint must produce visible progress.

A sprint is considered successful only when every planned feature is fully completed, tested, and documented.

Never leave partially finished systems.

---

# Sprint Duration

Recommended:

* 1 Week
* or
* 2 Weeks

Choose one duration and keep it consistent throughout the project.

---

# Sprint Structure

Each sprint should contain:

* Goal
* Features
* Tasks
* Priority
* Risks
* Deliverables
* Review

---

# Sprint Workflow

Planning

↓

Task Assignment

↓

Development

↓

Testing

↓

Bug Fixing

↓

Review

↓

Next Sprint

---

# Sprint Template

## Sprint Name

Example:

Sprint 01 — Foundation

---

## Goal

Describe the primary objective.

Example:

Build the First Person Controller and project foundation.

---

## Features

Example:

* First Person Controller
* Camera
* Input System

---

## Tasks

Each feature should be divided into smaller actionable tasks.

Example:

* Create Player object
* Configure movement
* Configure mouse look
* Configure collisions
* Test movement

---

## Priority Levels

High

Essential for game functionality.

Medium

Improves gameplay.

Low

Optional improvements.

---

# Definition of Done

A task is complete only if:

* Implemented
* Tested
* Stable
* Documented
* No critical bugs

---

# Sprint Review

At the end of every sprint answer:

* What was completed?
* What failed?
* What caused delays?
* What should improve next sprint?
* Were milestone objectives achieved?

---

# Sprint Rules

* Finish current sprint before starting another.
* Avoid adding new features during an active sprint.
* Keep sprint goals realistic.
* Prioritize quality over quantity.
* Remove blockers immediately.
