# ROOTS — RELEASE PLAN

## Purpose

This document defines how ROOTS moves from development to public release. A release is more than building the game; it is the final quality assurance process before players receive it.

---

## Release Philosophy

A release must be:

- Stable
- Tested
- Complete
- Documented
- Backed up

Never release unfinished features. Quality is always more important than release speed.

---

## Release Stages

### Stage 1 — Development

Features are actively being created.

Characteristics:

- Frequent changes
- Experimental systems
- High bug count

### Stage 2 — Internal Testing

Core features are complete.

Focus:

- Gameplay testing
- Bug fixing
- Performance
- Stability

### Stage 3 — Alpha

The game is playable from beginning to end.

Goals:

- Verify all major systems.
- Identify critical issues.
- Test progression.

Major features should already exist.

### Stage 4 — Beta

Feature complete.

Focus shifts to:

- Bug fixing
- Optimization
- Balancing
- User experience

No major systems should be added.

### Stage 5 — Release Candidate (RC)

Potential final version.

Requirements:

- No critical bugs
- Stable performance
- Complete documentation
- Final asset integration

Only critical fixes are allowed.

### Stage 6 — Version 1.0

Official release.

Requirements:

- Fully tested
- Stable
- Complete
- Ready for distribution

---

## Pre-Release Checklist

Before any release:

- Build succeeds.
- Save system verified.
- Story progression verified.
- Objectives verified.
- Dialogue verified.
- Audio verified.
- UI verified.
- Performance verified.
- Documentation updated.
- Backup created.

---

## Performance Checklist

Verify:

- Stable FPS
- Memory usage
- Loading times
- Scene transitions
- Save/Load speed

---

## QA Checklist

Confirm:

- No game-breaking bugs
- No missing assets
- No placeholder content
- No missing dialogue
- No broken objectives
- No inaccessible locations

---

## Documentation Checklist

Before release:

- Version History updated
- Bug Tracker reviewed
- Roadmap updated
- Milestones completed
- Task list synchronized

---

## Backup Policy

Before every release:

- Backup the Unity project.
- Backup documentation.
- Backup Git repository.
- Archive the release build.

Never release without a recoverable backup.

---

## Distribution Preparation

Prepare:

- Game icon
- Splash screen
- Screenshots
- Trailer
- Store description
- Changelog
- Version number
- License information

---

## Post-Release

Immediately after release:

- Monitor player feedback.
- Track reported bugs.
- Prioritize hotfixes.
- Plan the next update.

---

## Release Rules

A release is approved only if:

- All milestone goals are complete.
- Critical bugs = 0.
- High-severity bugs are reviewed.
- Performance targets are met.
- Documentation is complete.
- Producer approval is granted.

---

## Final Rule

A release represents the quality of the entire project. Never ship ROOTS because it is “good enough.” Ship it only when it reflects the quality standards defined throughout the project's documentation.
