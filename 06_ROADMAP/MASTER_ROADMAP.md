# ROOTS — MASTER ROADMAP

Version: 1.0

Status: Active

---

# PURPOSE

This roadmap defines the complete development lifecycle of ROOTS, from the first idea to post-release support.

Every AI Agent and contributor must follow this roadmap.

---

# PHASE 1 — PRE-PRODUCTION

## Goals

* Finalize game vision.
* Complete Master Bible.
* Finalize story outline.
* Define gameplay systems.
* Define technical architecture.
* Define art direction.
* Define audio direction.
* Create project documentation.

### Deliverables

* Master Bible
* GDD
* AI Agent System
* Project Rules
* Standards
* Roadmaps

Exit Condition:

Project scope is frozen and approved.

---

# PHASE 2 — PROTOTYPE

## Goals

Build a playable technical prototype.

### Systems

* Player Controller
* Camera
* Interaction
* Inventory
* Save / Load
* Pause Menu
* Settings
* Dialogue
* Evidence System

Exit Condition:

All core systems work together.

---

# PHASE 3 — PRODUCTION

## Goals

Create all game content.

### Content

* Levels
* Characters
* Props
* Documents
* Photos
* Audio Logs
* UI
* Story
* Puzzles
* Lighting

Exit Condition:

Entire game is playable.

---

# PHASE 4 — ALPHA

## Goals

Complete feature implementation.

Replace placeholders.

Integrate all assets.

Fix major issues.

Exit Condition:

Playable from beginning to end.

---

# PHASE 5 — BETA

## Goals

Bug fixing.

Optimization.

Polish.

Balancing.

QA.

Exit Condition:

Release Candidate approved.

---

# PHASE 6 — RELEASE

## Goals

Publish ROOTS.

Prepare launch assets.

Monitor launch quality.

Fix critical release issues.

Exit Condition:

Version 1.0 released.

---

# PHASE 7 — POST RELEASE

## Goals

Bug fixes.

Performance improvements.

Quality-of-life updates.

Community feedback.

Future planning.

# SUCCESS METRICS

* Stable 60 FPS.
* No Critical bugs.
* Story fully playable.
* Complete documentation.
* Consistent art style.
* Optimized build.
* QA approved.
* Producer approved.

---

# PROJECT RULES

Never expand scope without approval.

Never skip QA.

Never merge unfinished work.

Always document completed tasks.

Always follow project standards.

---

# MASTER RULE

Every task, feature and decision must move ROOTS one step closer to a stable, complete and release-ready game. If a task does not directly contribute to that goal, it should not be part of the project.
